Three interpretations, in order of likelihood given your context:

Event hook (internal trigger). Clean, testable, extensible. Webhook (external trigger). An outside system calls your app to tell it something happened. Example: Mesonet could webhook you when a rain event posts, and that inbound HTTP call triggers report generation. The catch — Mesonet almost certainly doesn't offer this, so you'd still be polling them. Webhooks are great when you own both sides or when the external service supports them. Probably not what you want here.
Cron/scheduler hook (time-based trigger). The "just a timer" from your note. On the server, a scheduled job runs (daily, hourly, whatever) and asks "what work needs to be done right now?" This is the dumbest and therefore most reliable option. Every real-world system of this kind — billing runs, email digests, backup jobs — is ultimately a timer underneath, no matter how fancy the abstraction looks.

Here's why. The auto-weekly report has a predictable cadence (once a week per project) and no external trigger that actually exists — Mesonet doesn't push to you, the "week rolled over" event is just "it's now a new week." The pattern that gives you both:

A single entry point called run_due_reports() that takes no arguments, queries the database for projects whose next scheduled report date is today or earlier, and generates reports for each. Pure function of database state.
A cron calls that function once a day. The cron does nothing but invoke the function — no logic lives in the cron itself.
That same function is also callable from a /admin/run-due-reports endpoint for manual triggering when you're testing or when a scheduled run fails and you need to catch up.
Tests call that function directly with a controlled database state. No cron needed in tests.
This gives you the "hook-like" feel you were reaching for — the generation logic is decoupled from the trigger — without buying a heavier abstraction than you need. The day you want to replace cron with something fancier (APScheduler, Celery Beat, a managed job queue), you swap the trigger layer and the core function doesn't change.

Tailwind CSS — it's the styling system your app already uses for its visual design. It came in during the original web migration we did way back. You don't need to be a frontend person to understand it; here's the gist.

The traditional way to style a webpage is to write a separate CSS file with rules like "all buttons should be blue with rounded corners and 12px padding," then your HTML just says <button class="primary-button"> and the CSS file decides what that looks like. Two files, two places to edit, separation of structure and style.

Tailwind flips that. Instead of inventing your own class names and writing CSS rules, you compose styling directly in the HTML using pre-made utility classes. So a button might look like <button class="bg-blue-600 rounded-lg px-3 py-2 text-white">. Each one of those is a tiny pre-defined CSS rule that Tailwind ships out of the box. bg-blue-600 is "background color: a specific shade of blue." px-3 is "horizontal padding: 12 pixels." And so on. You stack them like Lego bricks.

Why your app uses it: speed and consistency. You don't have to invent a design system from scratch; Tailwind's utility classes already encode sensible defaults for spacing, color, typography, and breakpoints. And critically, your app uses Tailwind without a build step — meaning you load it from a CDN (a script tag at the top of the HTML) and it just works. No compiler, no preprocessor, no npm run build. That was a deliberate choice during the web migration to keep the architecture simple.

The relevance to mobile responsiveness: Tailwind has built-in responsive breakpoints baked into every utility class. You can write <div class="grid-cols-1 md:grid-cols-2"> and that means "one column by default, two columns on medium-and-larger screens." The md: prefix is a responsive modifier. So when I say "the infrastructure is already in place for responsive design," that's what I mean — you don't need to install anything or restructure code. You just need to add the right sm: / md: / lg: modifiers to existing classes to tell each component how to behave at different screen sizes.

The teachable principle here is that frontend frameworks live on a spectrum from "do everything for you, learn our way" (Bootstrap, Material UI) to "give you primitives, you compose them" (Tailwind). The trade-off is that opinionated frameworks get you to a polished UI faster but everything ends up looking like the framework. Tailwind takes longer up front because you're making more decisions, but the final product looks like your product, not like every other Tailwind site. For a tool you're going to put your own brand on (sw3p.pro), that matters.

Bootstrapping a project — getting the initial scaffolding in place so further work has something to build on. Statistical bootstrapping — a resampling technique in stats. Different field, same metaphor.

It became dominant in the 2010s as the default way to make a website look professional without designing from scratch. If you ever used a WordPress theme that felt very "card-and-grid-and-blue-button" — clean rectangles, navbar at the top, footer at the bottom, the same general shape as half the web for a decade — there's a strong chance Bootstrap was underneath it. For your app, the choice was made during the original web migration — we picked Tailwind because the no-build-step approach (load it from a CDN, write utility classes in HTML, done) is dead simple to deploy and maintain.

Claude responded: Idempotent means: doing the operation more than once has the same effect as doing it once.
Idempotent means: doing the operation more than once has the same effect as doing it once.

The word comes from math (Latin idem "same" + potent "power") where it describes operations that don't compound when repeated. Multiplying by 1 is idempotent — 5 × 1 × 1 × 1 = 5. Multiplying by 2 is not — every repetition doubles you again.

In software, the question is: if I run this thing twice (or accidentally three times, or the network glitched and the same request hit the server four times), does the system end up in the same state as if I'd run it once?

A few examples to anchor it:

Idempotent: "set the user's name to Jake." Run it once, name is Jake. Run it ten more times, name is still Jake. Same end state.
Not idempotent: "add $50 to the user's balance." Run once, +$50. Run ten times, +$500. Wildly different end states. This is why payment systems are obsessed with idempotency — a duplicate webhook from Stripe should not charge the customer twice.
Idempotent: init_admin.py. Run it on a fresh database, it creates the admin invite. Run it again on a database that already has an admin, it sees the admin exists and exits without doing anything. Same end state either way: there is exactly one admin invite, and you can run the script as many times as you want without breaking anything. That's the design property I was pointing at.
Why it matters in practice: the real world is messy. Network requests time out and get retried. Cron jobs sometimes fire twice when a server hiccups. A user double-clicks the submit button. An agent runs your deploy script, isn't sure if it finished, and runs it again. If your operations are idempotent, none of this hurts you — at worst you waste a little CPU. If they aren't, you get duplicate charges, duplicate users, duplicate emails, corrupted state. Idempotency is one of the fundamental properties that separates "code that works on a sunny day" from "code that survives production."

The way idempotency is usually achieved: before doing the work, check whether it's already been done. The init script's logic is essentially "if no admin exists, create one; otherwise do nothing." That if check is the idempotency mechanism. In API design, the same idea shows up as "idempotency keys" — the client generates a unique ID for each request, the server remembers which IDs it's already processed, and silently ignores duplicates. Stripe's API does this; so do most modern payment processors.

The relevance going forward: when we design the auto-weekly report scheduler, idempotency is going to be a core requirement. If the cron fires twice on a Friday because of a server hiccup, we don't want two reports filed in the Mailbox for the same week. The function should check "has this week's report already been generated for this project?" and skip if yes. Same end state whether the cron ran once or five times. We'll bake that in from the start rather than discovering it the hard way.

It's one of those words that sounds intimidating until you have it, then becomes part of how you think about every system. Worth keeping in your back pocket.

Alpine.js
One sentence: Alpine.js is a small JavaScript framework that lets you add interactivity directly inside HTML attributes, without writing a separate JavaScript file or building anything.

The problem it solves: modern websites need to react to the user — clicking a button shows a dropdown, typing in a field updates a counter, toggling a switch hides a section. Doing that in raw JavaScript is painful: you have to give every element a unique ID, write JavaScript that finds the element, attaches a listener, manipulates other elements, keeps track of state. For complex apps you reach for React or Vue, which solve the problem properly but bring an entire build pipeline and a steep learning curve. For simple interactions, that's a sledgehammer for a thumbtack.

Alpine.js sits in the middle. It's about 15 KB of JavaScript loaded from a CDN — same script-tag-and-go pattern as Tailwind. Once it's loaded, you can sprinkle special attributes into your HTML and it makes them work.

A concrete example. Imagine a button that toggles a panel open and closed:

html
<div x-data="{ open: false }">
  <button @click="open = !open">Toggle</button>
  <div x-show="open">Hidden content here</div>
</div>
That's the whole thing. Three Alpine attributes:

x-data declares a little chunk of state — in this case, a variable named open that starts as false.
@click is shorthand for "when this element is clicked, run this code." Here, flip open to its opposite.
x-show means "only display this element when the expression is true."
There's no separate JavaScript file. No getElementById. No event listener wiring. The state lives right where it's used, in the markup. When open changes, Alpine automatically re-renders the parts that depend on it.

The philosophy is the same one Tailwind has for styling, applied to interactivity. Tailwind says "your styling lives in your HTML, in attributes, where the element is." Alpine says "your interactivity lives in your HTML, in attributes, where the element is." Both reject the older convention of "structure in HTML, styling in a CSS file, behavior in a JS file" — three separate concerns split across three separate files. The newer convention they share is "co-locate everything that defines a single component, in the place where the component lives." This is sometimes called the locality of behavior principle, and it's a meaningful philosophical break from the rigid separation-of-concerns thinking that dominated web development in the 2000s.

Where it sits relative to React/Vue: those frameworks are designed for full single-page applications — a website that loads once and then dynamically rebuilds itself as the user navigates, with no further page reloads. They're powerful but they replace your entire frontend architecture. Alpine is designed for "mostly server-rendered pages with sprinkles of interactivity" — exactly your situation. Your FastAPI backend renders HTML, Alpine handles the local interactions (toggling sections, validating fields, showing/hiding modals), and the page reloads when the user navigates between major surfaces. This is sometimes called the "HTML over the wire" approach and it's having a quiet renaissance after a decade of React eating everything.

Where you'll see Alpine in your codebase: every interactive element of the SWPPP form is wired up with Alpine. The checkboxes that change color when toggled, the rain-fetch button that disables itself while the request is in flight, the form sections that show validation errors, the project session selector — all of it runs on Alpine. Look at any HTML file under web/frontend/ and search for x-data, x-show, x-model, @click, @submit — those are the Alpine attributes doing the work.

Why it matters for our Phase 2 conversation: when we add features like the project selector, Mailbox toggles, mobile collapsible sections, automation settings switches — those are all Alpine work. The good news: Alpine is dead simple to extend. We're not learning a new framework, we're using more of what's already in the codebase. The bad news: Alpine is less battle-tested than React for very complex state. If a future feature has a lot of interconnected state (think: a multi-step wizard with dependencies between steps), we might hit Alpine's ceiling and need to either restructure or add a heavier framework. None of the Phase 2 features look like they'd push that limit, but it's worth knowing the boundary exists.

One last note worth tucking away: Alpine was created by an Australian developer named Caleb Porzio, originally as a Vue-inspired alternative for people who wanted Vue's reactivity model without Vue's complexity. It's open source, well-maintained, and used by a meaningful chunk of the "boring web" — sites built with frameworks like Laravel and Django that need interactivity without going full SPA. It's not a fashionable framework in the React-dominated discourse, but it's a smart, deliberate choice for what you're building. The original web migration picked it for exactly the right reasons.

Moving to question 3: failure recovery.

This is where the silent-failure hazard we flagged earlier gets its structural answer. Let me lay out the failure modes and what each one should do.

Failure mode A: Mesonet API is down or slow when the cron fires.

Expected. External dependencies fail. The cron shouldn't fail the whole daily run because one project's Mesonet lookup timed out.

Proposed behavior: For each project, catch Mesonet errors individually. If Mesonet fails for Project X, log the failure, mark Project X's last_run_status as mesonet_unavailable with a timestamp, continue to Project Y. At the end of the run, any projects in error status get retried on the next day's run. If Mesonet has been down for multiple days, the PM sees an increasingly urgent indicator on the project dashboard (green → yellow → red).

Failure mode B: PDF generation crashes on a specific project.

Template corruption, unexpected data shape, missing field — something specific to one project's content. Same principle: isolate the failure, log it, move on.

Proposed behavior: Catch generation errors per-project. Write a row to a run_failures table capturing project_id, error_type, error_message, stack trace, timestamp. Mark the project as generation_failed. Continue processing other projects. The PM sees the failure on the dashboard with enough context to act — "PDF generation failed for the template version active since 2026-04-10. Review the template or revert to the previous version."

Failure mode C: The server itself is down when the cron should fire.

Host went down, systemd crashed, a Tier 6 structured-log JSON formatter bug took out the whole service — whatever. The cron didn't fire at all on a given day. This is the big silent-failure risk.

Proposed behavior: Two layers of defense.

Self-healing via catch-up logic. When the cron does successfully fire, the first thing it checks for each project is "was I supposed to run yesterday (or the day before, or three days ago) and didn't?" If yes, it does the missed runs retroactively. A weekly report that was due Saturday but didn't run until Monday gets the Saturday date stamp, not the Monday date stamp. This is one of the most important design decisions in the whole scheduler — it means the system corrects its own gaps automatically when it comes back up.
External heartbeat. An outside monitor (probably an UptimeRobot or Healthchecks.io account, free tier) pings a /health endpoint on your server daily. If it doesn't hear back, it emails you. This catches the case where the server is down for multiple days and nobody notices — because an outside system is watching for us, not relying on us to notice our own failure.
Failure mode D: A new rain event occurred but Mesonet was down when we checked.

Compound of A and the rain-event logic. The rain-event check needs to be historically corrective — every daily run should look at the last, say, 14 days of Mesonet data for each project and compare against the Mailbox. Any rain-event day that doesn't yet have a report filed gets one generated now. So if Mesonet was down for three days and we missed a Wednesday rain event, we catch it on the next successful run.

This is where the idempotency pattern pays off again — "has a rain-event report already been filed for Project X on date Y?" is the check that prevents duplication. Same check that protects us from cron-firing-twice protects us from catch-up-logic-running-over-already-filed-data.

One architectural consequence worth naming: the run_due_reports() function isn't a simple "do today's work" loop. It's a reconciliation engine. Every time it runs, it asks "what's the desired state of the Mailbox for every active project, given today's date and history, and what's currently there? Fill in the gaps."

That's a stronger design than "run today's report." It's self-correcting by nature, tolerates outages gracefully, and has a single coherent mental model. It's also the pattern used by almost every production-grade scheduler in the industry — from Kubernetes controllers to Stripe's webhook delivery. You're landing on a good pattern not by accident but because the requirements pushed you there.

The PM's Home Screen — "My Projects"
When a PM logs in, the first thing they see is a list of their company's active projects. Not a fancy dashboard with graphs — just a list, one row per project, scannable in under three seconds. This is the air traffic control view.

Each row shows:

Project number and name. The identifier from the plans plus a human-readable label.
Status light. A single colored indicator that tells the whole story at a glance. Green = healthy, yellow = attention, red = problem.
Last report filed. "Friday, 4/18 — Weekly" or "Wednesday, 4/16 — Rain Event." The most recent Mailbox entry.
Next report due. "Friday, 4/25" for projects with auto-weekly on. "—" for manual-only projects.
Mailbox count. "23 reports" — a count, links to the project's Mailbox view.

The Status Light Logic
This is the clever bit. The rules that determine color should be strict enough to mean something but tolerant enough not to cry wolf.

Green: auto-weekly is off, OR auto-weekly is on and last_successful_run_at is within 24 hours of now. Also: no recent failures logged. Also: template was reviewed within the configured cadence window.
Yellow (attention): auto-weekly is on but last_successful_run_at is between 24 and 72 hours old. OR the template is approaching its review expiration (within 7 days of the cadence limit). OR there are non-critical warnings (Mesonet returned a partial response recently). Something to know about, nothing to act on urgently.
Red (problem): last_successful_run_at is more than 72 hours old. OR the last run logged a failure and the next run hasn't recovered. OR the template is overdue for review. OR more than three reconciliation runs in a week suggest the scheduler is struggling. Needs immediate eyes.

The Project Detail View — "The Health Panel"
Clicking a project row drills into the project detail page. Most of that page is the normal project workspace (settings, template, Mailbox, etc.), but one section is dedicated to run health:

Last run: timestamp, status (success/failed/partial), generation mode (scheduled/retroactive), and which report types fired (weekly, rain event, or both).
Run history: a simple table of the last 30 daily runs. Date, status, actions taken, notes if any. This is the audit trail for the PM's own use — "what has the scheduler been doing on my project?"
Open issues: any unresolved failures with action buttons. "PDF generation failed on 4/20 — template field missing. [View template] [Retry now]". The PM can manually retry a failed run without waiting for the next day's cron.

Two Small Architectural Notes Worth Flagging
First, the admin dashboard is a read-heavy page that aggregates across all tenants. That's the opposite of every other page in your system, which is scoped to a single company. Writing the queries correctly is critical. Specifically, any query on the admin dashboard that joins Projects to Companies needs to be deliberate — you want it to cross tenant boundaries. The isolation rules we're building into the app shouldn't apply to you in this view. This means the queries need to bypass the company-scoping middleware that other queries go through. Not hard, but it's a place where a mistake would either hide half the data from you (if you can't cross boundaries) or accidentally leak cross-tenant data to a regular user (if the boundary crossing isn't properly gated to admin-only).

The right pattern: a separate set of admin-only database queries that explicitly don't apply the company filter, guarded by a require_platform_admin dependency at the API layer. The queries themselves are simple; the access control is where the care goes.

Second, the admin dashboard data overlaps almost entirely with what the PM dashboard already computes, just aggregated differently. The temptation will be to write entirely separate code for admin views. Resist that — there should be a shared data layer that provides project health status, and both dashboards should consume from it. The difference is in the filter ("my company" vs. "all companies") and the aggregation ("show rows" vs. "show counts grouped by company"), not in the underlying data. This is more maintainable and guarantees the two views stay consistent as features evolve.

Company Onboarding Flow
The Handoff Model
Three parties, three handoffs:

Platform admin (you) → Company admin invite. You click "New Company" in your admin panel. You enter a company name and the primary admin's email address. The system generates a signup link with a token and emails it to that address. The link is one-time-use, expires in 14 days, and carries enough information that when the recipient clicks it they land on a pre-filled signup page.
Company admin claims the link. They click the link, land on a page that says "Welcome to sw3p.pro. You've been invited by [Your Name] to set up [Company Name]'s account." They fill out the company profile, set their password, and submit. That creates the Company record, creates their User record with company_admin role, and logs them in.
Company admin → employee invites. Now authenticated as the company admin, they use an internal invite UI to send codes (or links) to their employees. Each invite specifies a role (pm, inspector, viewer, or whatever role set we land on).
Each handoff is a bounded transaction. Each produces a specific data artifact. Each has a clear success state.

The Company Signup Form
When the new company admin claims their invite link, they see a form. Keeping this minimal is the right instinct — every field we add is a field someone has to fill out before they can use the product.

Required fields:

Company legal name. The name used in contracts, on invoices, on reports.
Display name. Same or different — what gets shown in the UI, on the Mailbox. Defaults to the legal name if blank.
Primary admin password. Set on signup (the invite carried the email).
Primary timezone. Default to something sensible based on location if we can infer, otherwise required dropdown. Used as the default for new projects.
Optional but recommended fields:

Company address. Mailing address. Shows up on some exports and eventually on invoices.
Company phone. Same reason.
Logo upload. For the Mailbox header and future branded PDFs. Can be skipped and added later.
Company website. For the Mailbox footer or company directory, if we ever build one.
Deliberately excluded:

Payment info. Billing is Phase 3. Signup should not require it.
DOT vendor ID, licenses, certifications. Compliance-type fields that vary by jurisdiction and could block signup. Add later if a real need surfaces.
Employee list. Onboarding your staff is a separate workflow, done after signup.

The User Record — Changes for Multi-Tenancy
Your current users table has is_admin as a boolean. That has to change.

New model:

users table: holds identity — id, display_name, email, password_hash, is_platform_admin, created_at, is_active.
company_users join table: holds company membership — user_id, company_id, role, joined_at, is_active.
A user can theoretically belong to multiple companies (rare but possible — a consultant who works for multiple GCs). A user is either a platform admin (you) or belongs to one or more companies (everyone else). Platform admins don't belong to any company via company_users; they're in their own category.

Migration path for your existing data: your current admin account (you) becomes is_platform_admin = true with no rows in company_users. Any test users you've created become members of "Company Zero" — your own GC's company, which becomes the first real tenant on the platform. Deletable or renameable later. Clean migration.

The Role Set
This is a decision with design consequences. Roles determine what a user can do inside their company. Here's a proposed minimal set:

company_admin — can manage company settings, invite/revoke employees, assign roles, create/edit/delete all projects, view the admin audit log.
pm — can create and manage their own projects, edit project settings, upload manual reports. Cannot manage company settings or other users.
inspector — read-only on all projects in the company. Can download reports from any project (not just the Mailbox — the logged-in view). Useful for office staff who need visibility but shouldn't be changing things.
viewer — extremely limited; essentially a dashboard observer. Might not be needed in Phase 2. Defer unless a real need emerges.
I'd actually suggest three roles for Phase 2 — company_admin, pm, inspector — and add viewer later if asked. Keep the permission model simple while we're learning what people actually need.

One thing to flag: PM-only project ownership is a design choice. The alternative is "any PM in the company can edit any project in the company." For Phase 2 I'd argue for the simpler model: any PM can see and edit any project. Most GCs are small enough that this isn't a problem, and per-project PM assignment adds real complexity (what happens when a PM leaves? Who reassigns? Can they reassign their own projects?). If tighter ownership becomes a need, add it later.
